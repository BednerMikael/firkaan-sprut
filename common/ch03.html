Администрирование ПС «СПРУТ»

Установка, настройка, управление пользователями и правами доступа, мониторинг, аудит.

IT-администраторУстановкаПрава доступаАудитOracle
Введение

Администрирование ПС «СПРУТ» — система действий по обеспечению функционирования ПО, управлению доступом пользователей, установке рабочих мест и архивированию данных.

⚠️
Для нормальной работы ПС «СПРУТ» требуется: локальная сеть TCP/IP, СУБД Oracle 8.0i+, при распределённой конфигурации — все серверы ЛБД доступны on-line по TCP/IP.
Раздел 1

Установка и настройка

1.1. Установка клиентской части BackOffice

Требования: ОС Windows, Oracle Client, BDE 5.0 с SQL Links к СУБД Oracle.

1
Запустите программу установки ПС «СПРУТ» → устанавливает в Program Files\Sprut. Компоненты: BDE 5.0, SQL Links к Oracle, все модули.
2
Настройка Windows Registry: укажите номер рабочего места (уникальный для каждого ПК в сети).
3
Настройка Oracle Client: прописать TNS-имя сервера БД, параметры соединения.
4
Настройка BDE: указать псевдоним БД с параметрами Oracle-соединения.
5
Настройка DOA: параметры прямого доступа к Oracle (отдельная библиотека).
6
DB LINK и DB LINK USER: для доступа к ЛБД — описать доступные ЛБД в Windows Registry.

1.2. Установка Сервера торговой площадки

Сервер ТП устанавливается отдельным инсталлятором на компьютер торговой площадки. После установки — настройка через модуль «Настройки сервера ТП» (см. Главу 30).

1.3. Установка СПРУТ POS

СПРУТ POS устанавливается отдельным инсталлятором на кассовый компьютер. При установке автоматически регистрируется внешнее устройство в основной БД.

Раздел 2

Подготовка справочников перед запуском

Перед началом работы необходимо заполнить базовые справочники:

Настройки параметров ПС «СПРУТ» — ключевые опции системы (тип НДС, округления, поведение модулей).
Список валют — национальная валюта и курсы иностранных.
План бухгалтерских и аналитических счетов — для экспорта в 1С и других бухгалтерских систем.
Правила — алгоритмы бухгалтерских проводок для различных типов операций.
Шаблоны документов — внешний вид и содержание печатных форм.
Регионы и менеджеры клиентов — структура отдела продаж.
Раздел 3

Управление пользователями и правами доступа

3.1. Профайлы и привилегии

Права доступа назначаются через Профайлы. Каждый пользователь привязывается к профайлу, который определяет его возможности в системе.

УровеньЧто ограничивает
МодулиКакие модули системы видит пользователь
ЖурналыКакие журналы внутри модулей доступны
Типы документовКакие типы документов может создавать/просматривать
СостоянияВ какие статусы может переводить документы
ДействияСпециальные операции: скидка сверх лимита, возврат без чека и т.д.
ОбъектыОтдельные поля и кнопки в интерфейсе
⚠️
Принцип минимальных привилегий: давайте пользователю только те права, которые необходимы для выполнения его работы. Лишние права — источник ошибок и злоупотреблений.

3.2. Порядок создания пользователя

1
Создайте профайл для роли (если ещё нет): Администрирование → Профайлы → «Добавить».
2
Назначьте привилегии профайлу: модули, журналы, типы документов, состояния, действия.
3
Создайте пользователя: Администрирование → Пользователи → «Добавить» → укажите логин, пароль, ФИО.
4
Привяжите пользователя к профайлу → сохраните.
5
Передайте логин и пароль сотруднику. Рекомендуйте сменить пароль при первом входе.
Раздел 4

Мониторинг работы пользователей

👥
Журнал «Сессии пользователей»
Кто, когда и с какого рабочего места входил в систему. Время начала и окончания сессии. Для контроля дисциплины и расследования инцидентов.
⚙️
Журнал «Контроль работы системы»
Системные события и ошибки. Ошибки подключения к БД, сбои при тиражировании, технические проблемы.
📋
Журнал «Аудит»
Все критичные изменения данных: кто, когда, что изменил. Фиксирует изменения в ключевых таблицах системы.
🔧
Очистка таблиц аудита: проводите регулярно (раз в месяц/квартал) чтобы БД не разрасталась. Предварительно экспортируйте аудит в архив если требуется долгосрочное хранение.
Раздел 6

Кэш заказов — особенности администрирования

При использовании кэша заказов (для ускорения работы с большим количеством заказов) необходимо:

Регулярно контролировать работу с кэшем через специальный модуль.
При сбоях — очищать кэш и перестраивать его.
Кэш заказов не является основным хранилищем данных — все данные хранятся в основной БД Oracle.
Раздел 7

Модуль «Кэширование метаданных» (srv_dictionary.exe)

Отдельный процесс, который кэширует метаданные системы (структуры таблиц, список полей, справочные данные) для ускорения загрузки модулей.

💡
При изменении структуры БД или обновлении системы — перезапустите srv_dictionary.exe для обновления кэша метаданных.
